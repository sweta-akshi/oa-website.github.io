Step 1 - Setting Up Knex and PostgreSQL

To add knex and postgresql to an Express application, first install the dependencies:

npm install knex
npm install pg

Next lets update our package.json with some helpful scripts:

package.json:

    "scripts": {
        "start": "NODE_ENV=production node server.js",
        "dev": "NODE_ENV=development PORT=3000 nodemon server.js",
        "db:create": "createdb knex_movies_dev",
        "db:drop": "dropdb knex_movies_dev",
        "db:reset": "npm run db:drop && npm run db:create",
        "db:migrate": "knex migrate:latest --env development",
        "db:seed": "knex seed:run --env development"
    },

We will be using these scripts in the following steps to create our database, create tables, and seed the database with data.

Step 2 - Creating the Database

For this lab we will create a simple database with no custom user/role. Simply run the following command to create the database:

npm run db:create

Then test it out with:

psql -d knex_movies_dev
psql (14.2)
Type "help" for help.

knex_movies_dev=# \q

Step 3 - Creating the Knex Configuration File

Knex uses a configuration file to connect to the database. The name for this file is usually knexfile.js.

Create this file with touch knexfile.js and then put the following contents inside the file:

const knexConfig = {
    development: {
        client: 'pg',
        connection: {
            host: '127.0.0.1',
            // user: "your_database_user",
            // password: "your_database_password",
            database: 'knex_movies_dev',
        },
        pool: {
            min: 2,
            max: 10,
        },
        migrations: {
            tableName: 'knex_migrations',
        },
        seeds: {
            directory: './seeds/dev',
        },
    },
}

export default knexConfig

Observations:

  • The above configuration sets up a single database for development. You can setup other databases for environments such as testingandproduction`.
  • The client defines which driver to use to connect to the database. Here we use the pg driver for connecting to a PostgreSQL database.
  • For this local database we will not need a user and password, but it depends on how you want to setup your database.
  • The pool setting defines the size for the DB connection pool.
  • The migrations and seeds settings configure how the migrations and seeds will work.
  • Review the knexfile.js Environment Configuration info for more information.

Step 4 - Database Migrations

Now that we have created our database and we have a knex configuration file, we can create our migration scripts.

npx knex migrate:make create_movies

The knex migrate:make command will create a migration script in the migrations folder. The name of the script is prefixed with a datetime stamp which knex uses to determine the correct order to run all of the migration scripts (oldest to newest).

Open the newly created migration script and edit it to have the following content:

export async function up(knex) {
    const exists = await knex.schema.hasTable('movies')
    if (!exists) {
        return knex.schema.createTable('movies', table => {
            table.increments('id').primary() // adds an auto incrementing PK column
            table.string('title').notNullable()
            table.string('genre')
            table.date('release_date')
            table.timestamps(true, true) // adds created_at and updated_at
        })
    }
}

export function down(knex) {
    return knex.schema.dropTableIfExists('movies')
}

Take some time to read this code and understand what each line is doing.

To run the migration scripts, simply use npm run db:migrate. Try this now to see if the above script runs successfully.

Creating the Knex Object

For our router file to use Knex, we must import the Knex module and then create a Knex object with the correct configuration:

Add the following to the routes/movies.js file:

import knexLib from 'knex'
import knexConfigs from './knexfile.js'

const knex = knexLib(knexConfigs[process.env.NODE_ENV])

There is a lot going on in these files so let’s break it down:

  • first we import the default export from knex and give it the name knexLib
  • then we import our configuration information from ./knexfile.js
  • finally we get the specific configuration (development, test, or production) from knexConfigs and pass it as a parameter to knexLib, giving us the knex object we need for executing knex commands.

Defining the Movies INDEX Route

The index route will simply return all of the movies from the database. Here is the code:

router.get('/', async (req, res, next) => {
    try {
        const movies = await knex.select('*').from('movies')

    } catch (err) {
        next(err)
    }
})

That was easy! All of our hard work in setting up Knex is finally paying off.

Observations:

  • We have wrapped all of our route handler code in a try/catch block.
  • If knex reports an error, it will result in a rejected promise that will be caught and reported to Express via the next callback.

Defining the Movies SHOW Route

The show route will return a single movie specified by the id URL parameter.

Add the following code to the routes/movies.js file:

router.get('/:id', async (req, res, next) => {
    try {
        const id = Number(req.params.id)
        const movie = await knex.select('*').from('movies').where({ id }).first()
        if (movie) {
            res.status(200).json(movie)
        } else {
            const error = new Error(`Movie with id ${id} not found.`)
            error.statusCode = 404
            next(error)
        }
    } catch (err) {
        next(err)
    }
})

Observations:

  • We are using an async function for our route handler.
  • We are using await with the knex statement.
  • We have wrapped all of our route handler code in a try/catch block.
  • If knex reports an error, it will result in a rejected promise that will be caught and reported to Express via the next callback.
  • We can get the value of the :id URL parameter from the req object via req.params.id.
  • The knex statement uses:
    • .select('*') to select all of the columns
    • .where({ id }) to return only the row with the matching id
    • .first() to return only the first row - there will only be one row but it will be inside an array so .first() unpacks it from the array.
  • If we don’t find the movie we report a 404 error.

Defining the Movies CREATE Route

The CREATE route will use an HTTP POST to receive JSON data defining a movie and INSERT it into the DB. Here is the code:

router.post('/', async (req, res, next) => {
    try {
        const { title, genre, release_date } = req.body
        const [saved] = await knex('movies').returning('*').insert({
            title,
            genre,
            release_date,
        })
        res.status(201).json(saved)
    } catch (err) {
        next(err)
    }
})

Observations:

  • We are using an async function for our route handler.
  • We are using await with the knex statement.
  • We have wrapped all of our route handler code in a try/catch block.
  • If knex reports an error, it will result in a rejected promise that will be caught and reported to Express via the next callback.
  • We destructure the req.body parameters into the title, genre, and release_date variables. This will only work if the json body-parser middleware is enabled.
  • We cannot use the knex .first() method with inserts and updates, but we can use JavaScript array destructuring to unpack the data returned from the insert() method.

Defining the Movies UPDATE Route

The UPDATE route will use an HTTP PUT to receive JSON data defining a movie and UPDATE it in the DB. Here is the code:

router.put('/:id', async (req, res, next) => {
    try {
        const id = Number(req.params.id)
        const movie = await knex.select('*').from('movies').where({ id }).first()
        if (!movie) {
            const error = new Error(`Movie with id ${id} not found.`)
            error.statusCode = 404
            next(error)
        } else {
            const { title, genre, release_date } = req.body
            const [saved] = await knex('movies')
                .where({ id: req.params.id })
                .returning('*')
                .update({
                    title,
                    genre,
                    release_date,
                })
            res.status(200).json(saved)
        }
    } catch (err) {
        next(err)
    }
})

Observations:

  • We are using an async function for our route handler.
  • We are using await with the knex statement.
  • We have wrapped all of our route handler code in a try/catch block.
  • If knex reports an error, it will result in a rejected promise that will be caught and reported to Express via the next callback.
  • Before executing the UPDATE statement, we check to see if the movie exists. If it isn’t found we report a 404 error.
  • We destructure the req.body parameters into the title, genre, and release_date variables. This will only work if the json body-parser middleware is enabled.
  • We cannot use the knex .first() method with inserts and updates, but we can use JavaScript array destructuring to unpack the data returned from the update() method.

We can test it out with:

http PUT localhost:3000/api/movies/7 title="Terminator 2: Judgment Day" genre="sci-fi" release_date="1991-07-03"

Defining the Movies DELETE Route

The DELETE route will use an HTTP DELETE to DELETE the specified movie from the DB. Here is the code:

router.delete('/:id', async (req, res, next) => {
    try {
        const id = Number(req.params.id)
        const movie = await knex.select('*').from('movies').where({ id }).first()
        if (!movie) {
            const error = new Error(`Movie with id ${id} not found.`)
            error.statusCode = 404
            next(error)
        } else {
        await knex('movies').where({ id: req.params.id }).del()
        res.status(200).json({ message: 'The movie has been deleted' })
        }
    } catch (err) {
        next(err)
    }
})

Observations:

  • We are using an async function for our route handler.
  • We are using await with the knex statement.
  • We have wrapped all of our route handler code in a try/catch block.
  • If knex reports an error, it will result in a rejected promise that will be caught and reported to Express via the next callback.
  • Before executing the DELETE statement, we check to see if the movie exists. If it isn’t found we report a 404 error.
  • We return a 200 status and a message wrapped in an object.
