Creating Mocks With Jest

Mocking is a powerful testing pattern use in just about all languages. It is often used to mimic dependencies of a function under test, so that the test can focus on the logic in the unit being tested.

Let’s say, for example, we want to test a middleware function that can be provided to an express route. An express middleware has the following method signature:

fn(req,res,next)

  • req short for request, is an object containing information about the incoming http request.
  • res is an object that has the supporting functions to respond to a client (res.send()) for example.
  • next is a callback function that is intended to be called if it is not sending back to the client.

So we have a module with the following function that can be provided as middleware:

const checkid = (req,res,next) =>{
    if (req.query.id){
        next()
    }
    res.status(413).send('{"message":"id is missing"')

}

module.exports = {checkid}

We have some behavior here that we want to test.

  • If the id is a valid param ( http://mydomain/?id="bob"), call next()
  • If it is not present, we want to an error code back to the client.

We don’t care how, or if req, res, or next work. We just need to be sure they are called under the correct conditions. So how do we do this without actually using an express route?

We create mocks that represent that will mimic the actual functions and objects passed to our function.

Manually Mocking

While we may not care in our test how req,res and next actually work, we should have an understanding of when to call them and how to use them.

Mock Req

const req = {
    query: {
        id: "bob"
    }
}

The real request object has a lot more in it. For our implementation, we really only care about the query.id property, so we can simply create a JavaScript object literal to represent the real request.

Mock next

let nextCalled = false
const nextFn = () => () => {
    nextCalled = true
}

next is a little more complex. However, in our code we just want to know that it was called at the appropriate time.

Setting nextCalled to true will give us a way to test that it was called at some point in the code.

Mock Res

const getRes = (rc) => {
    return {
        called: false, // 1
        code: 0,  // 1
        responseMessage: "", // 1
        status: function(c){  // 2
            this.code = c
            this.called = true
            return this // 3
        },
        send: function(message){ // 4
            this.responseMessage = message
        }
    }
}

Now things are getting pretty complex. However, this is still way more simple than the actual object.

  1. Rather than relying on a global variable, we will use a property of the object here to keep a record of our results.
  2. We know that we need a status function, so we define one here on the object. It will set the code property to whatever the calling code provides.
  3. We know that it is common to call res.status().send(), so to maintain that, we will return this. Returning this will allow us to chain the object. In other words we are able to chain a send call off of status because status returns the same object that has send in it.

We now have 3 mocks that will give us the behavior we expect when called, without calling the real thing.

Lets see what this looks like with tests.

const {
    checkid
} = require("./middleware")

// Manually creating mocks
let nextCalled = false
const nextFn = () => () => {
    nextCalled = true
}

const getRes = (rc) => {
    return {
        called: false,
        code: 0,
        responseMessage: "",
        status: function(c){
            this.code = c
            this.called = true
            return this
        },
        send: function(message){
            this.responseMessage = message
        }
    }
}
describe("#checkid", () => {
    describe("Given the checkid middleware", () => {
        describe("When the query param 'id' is present", () => {
            it("Should call next", () => {
                // Set up the default
                nextCalled = false
                const next = nextFn()

                // req can be a simple JSON object as the mock
                const req = {
                    query: {
                        id: "bob"
                    }
                }
                /* Get a mock our response object
                  and give it a return code to
                  so we can be sure the default 0 was
                  not overridden.
                */
                mockRes = getRes(123)

                //Make the actual call to our function with our mocks
                checkid(req, mockRes, next)

                // Ensure next was called
                expect(nextCalled).toBe(true)
                // Ensure no response was sent
                expect(mockRes.responseMessage).toEqual("")
                // Ensure no status code
                expect(mockRes.coding).toBe(0)

            })
        })
        describe("When the query param 'id' is NOT present", () => {
            it("It should respond with a 413", () => {
                // Set up/reset the default so we don't get a false positive.
                nextCalled = false

                // Get an instance of our mock next function
                const next = nextFn()

                // req can be a simple JSON object as the mock
                const req = {
                    query: {}
                }
                const mockRes = getRes()

                checkid(req, mockRes, next)

                expect(mockRes.coding).toBe(413)
                expect(mockRes.responseMessage).toEqual('{"message":"id is missing"}')
                expect(nextCalled).toBe(false)
            })
        })
    })

})

You do not need to create your mocks in the same file as your test. In fact, many teams may create their own modules for their own mocks to avoid rework.

  1. Create a file that exports a middleware function.
  2. Create your own mocks for res and next
  3. Write a tests that asserts they were called.

Using Jest to Create Mocks

Jest has built-in mocking library to help with creating mocks. There are some limitations, however. For example, we cannot easily auto-generate the same sort of mock object we built for response. We will convert our next function to demonstrate how this can lend to the readability of our code.

We can replace

let nextCalled = false
const nextFn = () => () => {
    nextCalled = true
}

with

const mockNext = jest.fn()

We can now use fn as a function

const mockNext = jest.fn()

checkid(req, mockRes, mockNext)
expect(mockNext.mock.calls.length).toBe(1);

In short, jest gives us the ability to quickly create mock functions that will allow us to easily validate without having to write extra validation variables to determine if the function was called.

Mocking tools can be powerful, but sometimes hard to grasp. Here are ust a few more quick things you can do with Jest function mocking that you may find useful and easy to use right away:

More examples of mocking with Jest

//Create your own implementation behavior.
const mathMock = mock.fn(x=>2+x)
console.log(mathMock(5))

// This can even be used for your asserts
const fnAndExpect = mock.fn(()=>expect(true).toBe(true))
fnAndExpect()

// Have prepared return values
const mockDataFetch = mock.fn(()=>'{"data":0101}')
console.log(mockDataFetch())

// Chain prepared return values
const mockChinReturn = jest.fn()
    .mockImplementationOnce(() => 1)
    .mockImplementationOnce(() => 2)
    .mockImplementationOnce(() => 3)

console.log(mockChinReturn())
console.log(mockChinReturn())
console.log(mockChinReturn())

It is worth at least skimming over the other functions that Jest offers in its mocking library. You can find them here

Another powerful mocking library that can be used with any testing framework is SinonJs, commonly referred to as just Sinon. In addition to mocking it has powerful server testing utilities built in so that you can easily manage starting/stopping servers if needed.

Testing Express

Just like any other code, we want to be able to test the behavior of our endpoints.

Lets say we have the following code:

const express = require('express')
const app = express()

app.get('/endpoint', (req, res) => {
    res.status(200).send('It worked!')
})

module.exports = app

We need a way to unit test that the response function was called appropriately when the /endpoint path was called.

Still using Jest, we can use a very popular library called supertest.

Supertest

Supertest is library that extends another popular library called superagent which will allow us to mock client call to ensure our responses are working as expected.

Installing Supertest

As with jest, we want to install the supertest module as a dev dependency. Therefore, be sure to use the -D flag when installing the module from npm.

npm install -D supertest

Once installed, your devDependencies section should contain at least the following, at whatever the current versions are:

  "devDependencies": {
    "jest": "^23.6.0",
    "supertest": "^3.3.0"
  }

Your first test using supertest

The supertest library allows us to write tests in a format that is similar to other client libraries that make calls to endpoints, such as request-promise or axios.

const request = require('supertest'); // 1
const routes = require('./app')  // 2

describe('Get /endpoint', () => { // 3
    it('It should response the GET method', () => {

        return request(routes).get('/endpoint').then((response) => {
            expect(response.statusCode).toBe(200);
        }); // 4

    });
});
  1. Import supertest. Although not required, it is common practice to call the import request
  2. Import your routes. For unit tests, it is important not to import your code that invokes the listen(fn()) function from express
  3. Describing your route under test, as well as the HTTP method is not required, but results in a more readable output.
  4. Supertest returns a promise after mocking the appropriate call specified after the request function. In this case, we make a GET call to the /endpoint

Built in Assertions

In addition to making the request, supertest also provide some common assertions you’d make with http calls.

For example, we could covert the above test to use built the built in expect function:

        return request(routes)
        .get('/endpoint')
        .expect(200);

When using the built in expect assertion, the assertion message is slightly different.

Rather than seeing:

Expected: 200 Received: 404

You will see:

expected 200 “OK”, got 404 “Not Found”

You can still use then to do further assertions. supertest will resolve a response object that will contain a lot of the things you’d expect when making a client call in node.

For example, if we wanted to assert the response text of a call:

    return request(routes).get('/endpoint')
    .expect(200)
    .then((response) => {
        expect(response.text).toBe("Hello");
    });

In this case we first expect a 200, then expect the text to be equal to Hello

When chaining built-in expects, the promise will return at the first failure.

Setting Request Headers

It is common for clients to set specific headers, such as telling the server it would like an 'application/json' response.

    return request(routes).get('/endpoint')
            .set('Accept', 'application/json')
            .expect('Content-Type', /json/)
            .expect(200)
            .then((response) => {
                expect(response.body).toEqual({message: "hi"});
            });

You can test all http methods with supertest.

Designing Testable Routes

Since we want to focus on only the code within a given unit for unit testing, we want to ensure that is ALL we are testing.

For express routes, we don’t necessarily want to test the entire middleware chain for a unit test. We also don’t want to make any real calls to external resources. However, we do still expect the routes to give a specific response when called.

Injecting Middleware

A common piece of middleware used in express routes is some sort of authorization middleware. For example, lets say we are securing the following endpoint:

app.get('/secure/endpoint', isAuthorized, (req,res) =>{
    res.status(200).send("All OK!")
} )

Lets assume that isAuthorized is a middleware function that communicates with some sort of authorization service. However, when we test the /secure/endpoint route, we don’t care specifically what this is doing. So how can we make this code testable with this isAuthorized external dependency?

There are a few methods that can be used to inject your dependencies.

Factory Function

A factory function is a way to implement the factory pattern. The factory pattern is simply a way to generate instances of something in a specific configuration with a single function.

We will refactor the route above to implement a factory function to allow for dependency injection.

const express = require('express')

const secureEndPoint =  (authMiddleware)=>{
    const router = express.Router();
    router.get('/secure/endpoint', authMiddleware, (req,res) =>{
        res.status(200).send("All OK!")
    } )
    return router
}

module.exports = {secureEndPoint}

Our factory function is called secureEndPoint the instance of the “thing” we are returning is a configured express router.

Now to test this, we can create our own mock and pass it to the factory function as part of the test setup.

const request = require('supertest');
const {secureEndPoint} = require('./app')
const express = require("express")
let app

// Initialize "app" before each test
beforeEach(()=>{
    app = express()
})

// Mock auth object.
const mockAuth = (req,res,next) =>{
    next()
}

describe('GET /secure/endpoint', () => {
    it('It should respond with a 200', () => {
       // Use app.use to add the route
       app.use(secureEndPoint(mockAuth))

       // Test as normal
       return request(app)
        .get('/secure/endpoint')
        .expect(200)
    });
});

Not only have we tested, but we also have documentation on how the route is intended to be configured and used.

You can set up multiple routes at a time this way. For example, say you have a group of endpoints that you want to secure in this manner. You could add as many routes that need the same decency as you want in the same function.

Testing Routes that call external services.

You will run into situations where dependency injection won’t make anything more testable. In fact, it can make things harder to test and read if you attempt to.

IO tends to be one of the major areas. In a lot of cases, this is where you can draw the line and commit to covering the code in an integration test.

However, there are some tools and methods you can use to test routes that use calls to services.

Nock

Lets say our secure endpoint makes a call to dapper:

const secureEndPoint = (authMiddleware) => {
    const router = express.Router();
    router.get('/secure/endpoint', authMiddleware, (req, res) => {
        request('http://dapper.apps-np.homedepot.com/users/exw5373')
            .then(json => res.json(json))
    })

    return router
}

There is no real way we can inject the request here. If the network is down or the site is down, then our unit tests fail. That is not an acceptable failure reason for a unit test.

Nock allows us to intercept calls and reply with mock responses. It works by overriding the http.request

Using nock in our test, we are able to intercept it an form our own reply:


    /*
    1. Provide the domain we need
    2. Specify a get the expected path
    3. reply with the status code and message

    */
   nock('http://dapper.apps-np.homedepot.com')
   .get('/users/exw5373')
   .reply(200,{message: "you got nocked!"})

    it('It should respond with a 200', () => {
       app.use(secureEndPoint(mockAuth))
       return request(app).get('/secure/endpoint').
       expect(200)
       .then(res => {
            console.log(res.body)
       })
    });

If you set up a nock, it must be used or the test fails.
