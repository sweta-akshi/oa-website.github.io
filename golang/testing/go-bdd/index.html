Behavioral Driven Development

Behavioral Driven Development (BDD) is a software development approach that has evolved from TDD (Test Driven Development). In both development approaches, tests are written ahead of the code, but in BDD, tests are more focused on the behavior of the code rather than a specific implementation.

Tests are often written with the following thought process:

Gherkin style

Given a specific context
When an action is taken
Then this is the expected result

Given 2 and 2 as input to Add
When the function is invoked
Then it should return 4

Ginkgo

Ginkgo is a BDD-style Go testing framework built to help you write expressive tests. In the following examples, we will use Ginkgo with the Gomega assertion library but can be used with other assertion libraries.

Installing Ginkgo and Gomega

Just go get it:

go get github.com/onsi/ginkgo/ginkgo
go get github.com/onsi/gomega/...

This will install both ginkgo and gomega and place the executables in $GOPATH/bin and your go.mod (if using Gomods).

To see if you successfully installed ginkgo, run ginkgo version. You should get something like:

Ginkgo Version 1.8.0

Ginkgo Test Suite

Ginkgo works directly with Go’s existing testing library. This means that you can run a Ginkgo suite using go test.

To create Ginkgo tests for a package you must first generate the code for a Ginkgo test suite.

For our examples, we will be using a lumber package, with the following lumber.go file:

package lumber

import "fmt"

type Lumber struct {
	WoodType 	string
	Price 		float64
	Width 		int
	Height 		int
	Length 		int
}

//LumberArea calculates the area of the lumber in inches
func (l Lumber) LumberArea() int {
	return l.Width * l.Height * l.Length;
}

//PricePerSqInch finds the price per square inch of lumber
func (l Lumber) PricePerSqInch() float64 {
	return l.Price / float64(l.LumberArea());
}

//Cut removes x number of inches. Returns an error if x > lumber length.
func (l *Lumber) CutUp(x int) error {
	if(x < l.Length){
		l.Length -= x
		return nil
	}
	return fmt.Errorf("%v is too long", x)
}

Example: to generate tests for the lumber package:

cd path/to/lumber
ginkgo bootstrap

This will create a file named lumber_suite_test.go with the following contents:

package lumber_test				//1

import (
	"testing"

	. "github.com/onsi/ginkgo"  //2
	. "github.com/onsi/gomega"  //2
)

func TestLumber(t *testing.T) {	//3
	RegisterFailHandler(Fail)	//4
	RunSpecs(t, "Lumber Suite")	//5
}
  1. Ginkgo sets up the test with the lumber_test package, instead of lumber allows us to respect the encapsulation of the lumber package. You can, of course, opt out of this – just change package lumber_test to package lumber.
  2. By default, both Ginkgo and Gomega packages are dot imported. Dot imports bring the package into the same namespace as your current package, so you no longer have to use the imported package’s name to reference its variables and functions, giving us direct access to them.
  3. TestLumber is a testing test. The Go test runner will run this function when you run go test or ginkgo.
  4. RegisterFailHandler is a Gomega function that is taking in a Ginkgo function called Fail(description string). Fail is triggered when a test fails.
  5. RunSpecs(t *testing.T, suiteDescription string) tells Ginkgo to start the test suite. Ginkgo will automatically fail the testing.T if any of your specs fail.

At this point you can run your suite with either ginkgo or go test:

Watching For Changes

Placing ginkgo watch in the terminal starts constant monitoring of the package in the current directory. Any change to the package triggers the tests.

For each monitored packaged, Ginkgo will also monitor that package’s dependencies and trigger the monitored package’s test suite when a change in a dependency is detected. By default, ginkgo watch monitors a package’s immediate dependencies.

Adding Specs

It is possible to add tests directly in the file generated above, but it is preferred that your tests are in separate files.

To create a test file for the lumber package, run the command ginkgo generate lumber in the lumber directory in the terminal.

This will create a file called lumber_test.go that has the following contents:

package lumber_test

import (
	. "github.com/onsi/ginkgo"
	. "github.com/onsi/gomega"

	. "github.homedepot.com/lumber"
)

var _ = Describe("Lumber", func() {

})

We add a top-level describe container (defined below) using Ginkgo’s Describe(text string, body func()) bool function. var _ = ... allows us to evaluate the Describe at the top level without having to wrap it in a func init() {}

Organizational functions

Below we have updated lumber_test.go to have three tests for the LumberArea, PricePerSqInch, and CutUp methods.

var _ = Describe("Lumber", func() {		//1

	Context("With (W, H, L) = (5, 2, 60)", func() {			//2
		It("should be have area of 600", func() {			//3
			myLumber := Lumber{
				WoodType: "pine",
				Price: 12.75,
				Width: 5,
				Height: 2,
				Length: 60,
			}
			Expect(myLumber.LumberArea()).To(Equal(600))	//4
		})

		It("should be $0.02 per square inch", func() {		//2
			myLumber := Lumber{
				WoodType: "pine",
				Price: 12.75,
				Width: 5,
				Height: 2,
				Length: 60,
			}
			actual := myLumber.PricePerSqInch()				//3
			Expect(actual).To(Equal(0.02))					//4
		})

		It("should shorten the lumber 2 inches", func() {	//2
			myLumber := Lumber{
				WoodType: "pine",
				Price: 12.75,
				Width: 5,
				Height: 2,
				Length: 60,
			}
			myLumber.CutUp(2)				//3
			Expect(myLumber.Length).To(Equal(58))	//4
		})
	})
})
  1. Describe not only verifies the tested code, but also documents its intended behavior. Each Describe block specifies a larger component or function and contains a set of specifications.
  2. Context defines a set of tests that test related functionality under one state. Usually you will have one context per file, but you may have multiple contexts in a single file if you so choose. There is no functional difference between Context and Describe. However, there is a contextual difference:
  • Describe wraps a set of tests against one functionality.
  • Context wraps a set of tests against one functionality under the same state.
  1. It defines the specifications. Each It block functions as a test and is evaluated in its own environment.
  2. Expect is Gomega assertion function takes in an actual assertion and is followed by a chain of qualifications. To takes in an Gomega function. Equal is a function that uses deep equality instead of strict equality to compare actual with expected.

It Aliases

In order to ensure that your specs read naturally, Ginkgo supplies aliases for It: FIt, PIt, XIt, Specify, FSpecify, PSpecify, XSpecify.

There are corresponding Specify blocks available as aliases to use in situations where the corresponding It alternatives do not seem to read as natural language. All the same protocols apply for Specify blocks which apply to It blocks.

Equivalencies

  • It corresponds with Specify
  • FIt corresponds with FSpecify
  • PIt corresponds with PSpecify
  • XIt corresponds with XSpecify

PIt

PIt (or XIt) marks individual Its as temporarily pending, and not execute any blocks that are inside that block.

The output will not reflect if the specified test is failing or passing. A good case to use this is when refactoring code that breaks tests, to avoid deleting or commenting-out that hard-won test logic while you get back to green, test by test.

By Function

Generally you should try to keep your Its short and to the point. This is not always possible, however, especially in the context of integration tests that capture a particular workflow. You can capture these workflows with By.

By must be called within a runnable block (It, BeforeEach, etc…) By will simply log the passed in text to the GinkgoWriter. If By is handed a function it will immediately run the function.

var _ = Describe("Lumber", func() {
	Context("With (W, H, L) = (5, 2, 60)", func() {
		It("should be have area of 600", func() {
			myLumber := Lumber{
				WoodType: "pine",
				Price: 12.75,
				Width: 5,
				Height: 2,
				Length: 60,
			}
			Expect(myLumber.LumberArea()).To(Equal(600))

		It("should be $0.02 per square inch", func() {
			myLumber := Lumber{
				WoodType: "pine",
				Price: 12.75,
				Width: 5,
				Height: 2,
				Length: 60,
			}
			actual := myLumber.PricePerSqInch()
			actual = math.Round(actual*100)/100
			Expect(actual).To(Equal(0.02))
		})

		It("should shorten the lumber 2 inches", func() {
			myLumber := Lumber{
				WoodType: "pine",
				Price: 12.75,
				Width: 5,
				Height: 2,
				Length: 60,
			}
			By("Checking new lumber length")	// Adding additional statement for failing tests
			err := myLumber.CutUp(2)
			Expect(myLumber.Length).To(Equal(58))
			By("Checking that there was not an error thrown")	// Adding additional statement for failing tests

			Expect(err).To(BeNil())
		})

Suites

Sometimes you want to run some set up code once before the entire test suite and some clean up code once after the entire test suite. For example, perhaps you need to spin up and tear down an external database.

Ginkgo provides BeforeSuite and AfterSuite to accomplish this. You typically define these at the top-level in the bootstrap file

We can update lumber_suite_test.go to use both BeforeSuite and AfterSuite


var dbRunner *db.Runner	// Mock database runner
var dbClient *db.Client	// Mock database client

func TestLumber(t *testing.T) {
	RegisterFailHandler(Fail)
	RunSpecs(t, "Lumber Suite")
}

var _ = BeforeSuite(func() { //1
		By("Connecting to database")
    dbRunner = db.NewRunner()
    err := dbRunner.Start()
    Expect(err).NotTo(HaveOccurred())

    dbClient = db.NewClient()
    err = dbClient.Connect(dbRunner.Address())
    Expect(err).NotTo(HaveOccurred())
})

var _ = AfterSuite(func() { //2
		By("Disconnecting from database")
    dbClient.Cleanup()
    dbRunner.Stop()
})
  1. Will be run before all tests in the Lumber Suite. Connecting to our mock database and making sure there was not an error connecting to our mock database.
  2. Will be run after all tests in the Lumber Suite. Clean up and disconnect from our mock database.
