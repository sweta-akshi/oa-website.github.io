Where to Start

Inside Out vs Outside In

Where to start testing can sometimes be a bit hard to determine. There are two approaches that work, whether you are using TDD or not. These are strategies are known as: Inside Out and Outside In.

Inside Out: This method focuses on testing and building the inner components first. Business logic, data retrieval and other components that are well defined can be created. From there, you work your way out to the interface. In our case, the endpoint/route that clients will call.

Outside In: Outside in starts by designing and testing the API endpoints, and working your way in as you go.

We can use these approaches as options in our toolbox of testing techniques to help narrow down where we should start. Each has an advantage in certain situations. First and foremost, what method works for you and/or your team is how you should start. But we can take a look at some basic guidelines of when each applies.

When to Use Inside Out

You might chose Inside Out if you have:

  • A well designed API already in place
  • Complex business logic inside of your application
  • Data that needs to be retrieved from many different sources before it is ready to be used by an external application
  • You are unsure of how your data will be accessed, but know you will need certain components.

Pros

  • Can allow you to tackle more complex parts of your application first.
  • Can providing reusable components without being tied to a specific user interface
  • Code to architecture requirements

Cons

  • Potentially develop components that won’t be needed
  • Possible to lead to tightly coupled architecture.

When to Use Outside In

You might chose Outside In if:

  • You are working with a brand new API
  • You are unsure of how the component interaction will be
  • Delivering an MVP that may not have all components ready
  • You are unsure of how your data will be accessed, but know you will need certain components
  • You have a strong understanding of your user or application interaction with your application

Pros

  • Can deliver something truly usable, even if not fully ready almost from the start
  • Ensures you are only developing components that you will need

Cons

  • Narrow focus on the user interface
  • Good design patterns and usage for internal components are not always apparent.
  • May lock users into specific usage of your application before understanding the big picture.

Use Both

Don’t feel stuck to a single approach. As you learn your application better, you will be able to more easily switch between the two when it makes sense. Again, they are just additional tools to help you develop great software. In the end, you still need to deliver value.

In this course, we will start with testing our handlers and work our way out. From there, you can choose either approach.

The httptest Package

The httptest package is a simple, but powerful, built-in package that provides a few types and functions to assist with testing almost all types of http functionality.

  • NewRequest function will generate an http request to be used in our handlers
  • ResponseRecorder provides us with a data structure that can be used to test the behavior of a handler. In addition to providing methods to assist in testing, the ResponseRecorder implements the ResponseWriter interface so it can be used in your handlers as well.
  • Server provides a data structure and methods for testing your endpoints end to end and/or integration testing. It can be used to easily spin up a server and listen to a specified port on the loopback interface.

Testing the http.Handler

We’ve been asked to provide a handler that will be used for a complex mathematical equation. The ask is that we take any number equal to 100 or below and determine if it is odd or even. Some additional requirements were provided:

  • If the number is 101 or above, then we want to return an HTTP status code of 406
  • If the number is 0 or below, then we want the server to blow up and return a 500 and indicate that math is hard.
  • All responses should return JSON with the appropriate headers set.

Great, now we have some criteria defined for lets review what we have to work with.

First, review the signature of the http.Handler function.

func(w http.ResponseWriter, r *http.Request)

Based on this we know we need to mock, or somehow create a ResponseWriter and a pointer to an http.Request. For the ResponseWriter you could implement the interface with a MockResponseWriter yourself. That may in fact be a solution for some one off situation you find yourself in. However, as we were introduced to the httptest package we are already provided with the tools we need to meet the dependencies.

Let’s get our test set up:

package main

import (
    "net/http"
    "net/http/httptest"

)

func TestEvenOrOddHandler(t *testing.T) {
    /*
        Creates a new request object of type GET. Note your handler 
        probably doesn't care about this method and you could put anything
        you wanted to as long as it is a string in this position for this test.
    */
    req := httptest.NewRequest("GET", "/evenorodd?number=2", nil)

    // Gets me a ResponseWriter
    w := httptest.NewRecorder()
}

 

Now if we look at our handler we can see we just need to invoke it with the two values:

func EvenOrOddHandler(w http.ResponseWriter, req *http.Request) {
    // At some point in the TDD Process we'd add better code to ensure values were OK
    if num, ok := req.URL.Query()["number"]; ok {

        n, _ := strconv.Atoi(num[0])

        r := "odd"
        if n % 2 == 0 {
            r = "even"
        }

        fmt.Fprintf(w, `{"result:" "%s"}`, r)
    }

    fmt.Println()
}

 

The test:


func TestEvenOrOddHandler(t *testing.T) {
        req := httptest.NewRequest("GET", "/evenorodd?number=2", nil)
        w := httptest.NewRecorder()

        EvenOrOddHandler(w, req)
        assert.Equal(t, w.Code, 200)
        assert.Equal(t, w.Body.String(), `{"result:" "even"}`, "I Should be even!")
}

 

 

As you can see, we can use Code and Body properties of the ResponseRecorder struct that we got from NewRecorder to validate we got the expected results.

Now that we have some basic test criteria working, we can include some other input tests. At this point we’ll go ahead and introduce a table driven test and refactor our function to make everything pass. Here is the entire code, including the handler that we tested:

 

package main

import (
        "fmt"
        "net/http"
        "net/http/httptest"
        "strconv"
        "testing"

        "github.com/stretchr/testify/assert"
)

func TestEvenOrOddHandler(t *testing.T) {
        tests := []struct {
                name         string
                message      string
                expectedCode int
                queryParam   string
                expected     string
        }{
                {
                        name:         "Test an even number below 100",
                        message:      "It should return 200 and be even",
                        expectedCode: 200,
                        queryParam:   "2",
                        expected:     `{"result:" "even"}`,
                }, {
                        name:         "Test an equal to 100",
                        message:      "It should return 200 and be even",
                        expectedCode: 200,
                        queryParam:   "100",
                        expected:     `{"result:" "even"}`,
                },
                {
                        name:         "Test an odd number below 100",
                        message:      "It should return 200 and be odd",
                        expectedCode: 200,
                        queryParam:   "99",
                        expected:     `{"result:" "odd"}`,
                },
                {
                        name:         "Test a number over 100",
                        message:      "It should return 406 and be error",
                        expectedCode: 406,
                        queryParam:   "121",
                        expected:     `{"result:" "error"}`,
                },
                {
                        name:         "Test a number bellow 0",
                        message:      "It should return 500",
                        expectedCode: 500,
                        queryParam:   "-1",
                        expected:     `{"result:" "math is hard"}`,
                },
        }

        for _, test := range tests {
                t.Run(test.name, func(t *testing.T) {
                        req := httptest.NewRequest("GET", fmt.Sprintf("/evenorodd?number=%s", test.queryParam), nil)
                        w := httptest.NewRecorder()

                        EvenOrOddHandler(w, req)
                        assert.Equal(t, test.expectedCode, w.Code)
                        assert.Equal(t, "application/json", w.Header().Get("Content-Type"), "application/json")
                        assert.Equal(t, w.Body.String(), test.expected, test.message)
                })
        }

}

func EvenOrOddHandler(w http.ResponseWriter, req *http.Request) {
        // At some point in the TDD Process we'd add better code to ensure values were OK
        if num, ok := req.URL.Query()["number"]; ok {
                n, _ := strconv.Atoi(num[0])

                r := "odd"
                if n > 100 {
                        w.WriteHeader(http.StatusNotAcceptable)
                        r = "error"
                } else if n <= 0 {
                        w.WriteHeader(http.StatusInternalServerError)
                        r = "math is hard"
                } else if n%2 == 0 {

                        r = "even"
                }
                w.Header().Add("Content-Type", "application/json")
                fmt.Fprintf(w, `{"result:" "%s"}`, r)
        }

        fmt.Println()
}

 

Testing Gorilla/Mux Middleware

So now we’ve delivered a component that is able to run the most complex mathematical algorithm in history. Every one is super excited. Demo and retro went well.

However, the customer is now super concerned about who has access to this immensely powerful tool that your team has created. They don’t want any malicious Spring Boot apps utilizing this service for dastardly deeds. Luckily we know we have a authentication service and middleware we can use for just such a thing. We just need to test it.

Testing middleware will be much like testing our handlers. We need to slightly alter our expectations. Middleware can respond to the client as a normal handler would, or it can invoke the “next” handler in our middleware chain.

Because we have an awesome PM that helped get wonderful acceptance criteria, we know we need:

  • If the user is authorized, call the next handler.
  • If the user is NOT authorized respond with a 403

Right now our 0% code covered code looks like:

func Auth(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, req *http.Request) {

        user := req.URL.User.Username()
        pass, _ := req.URL.User.Password()
        if superComplexAuthenticationAndAuthorizationLogic(user, pass) {
            req.Header.Add("Authorized", "Sure is")
            next.ServeHTTP(w, req)
            return
        }

        w.WriteHeader(http.StatusUnauthorized)
        fmt.Fprintf(w, "")
    })
}

 

 

In our production code we will be passing Auth to the mux.Use() method. In our test we just want to be sure that our function behaves as it should when the Gorilla framework invokes it. So we will need to come up with a way to verify that next gets invoked OR 403 gets set as the response code.

For our test we want to implement the handler interface with a mock/stub:  


type MockHandler struct {
    t              *testing.T
    Message        string
    wascalled      bool
}

func (mh *MockHandler) ServeHTTP(w http.ResponseWriter, req http.Request) {

    // We could assert other things here, but for now we just 
    // want to know if it was called
    mh.wascalled = true

The test runs great. However, as you start advancing through the test you notice a big problem. You are calling the real service each time. This means to test you are forced to test you are forced to provide real users and passwords, or set up a special configuration that points to a test auth service. Not only do we don’t have time for that, we can’t unit test this code because it is not unit testable.

We know we somehow need to replace the functionality of superComplexAuthenticationAndAuthorizationLogic. We need a way to inject this dependency into our code. So we need to do a bit of refactoring. There are a few valid ways we could approach this.

For example we could refactor Auth to take in a function that returns the same the middlewarefunction:  

func Auth(auth func(string,string) bool) func(http.Handler) http.Handler

 

We won’t bother displaying the rats nest of returned functions that this would end up being. While totally valid, and just as flexible/reusable as other methods, we start to lose readability.

So what if, instead, we were to create a struct that allowed for the service to be set as a property, and then just attach our Auth middleware to it?

 

type AuthMiddleware struct {
        AuthService func(string, string) bool
}

func (am AuthMiddleware) Auth(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, req *http.Request) {

                user := req.URL.User.Username()
                pass, _ := req.URL.User.Password()

                if am.AuthService(user, pass) {
                        req.Header.Add("Authorized", "Sure is")
                        next.ServeHTTP(w, req)
                        return
                }

                w.WriteHeader(http.StatusUnauthorized)
                fmt.Fprintf(w, "")
        })
}

 

Now do a slight refactor to our test code:

        as := AuthMiddleware{
                AuthService: func(user, pass string) bool {
                        return test.isAuthed
                },
        }
        as.Auth(mh).ServeHTTP(w, req)

func TestAuth(t *testing.T) {
        tests := []struct {
                name     string
                message  string
                user     string
                pass     string
                isAuthed bool
                callnext bool
                rc       int
        }{
                {
                        name:     "Given the User was Authorized",
                        message:  "The next handler should be called",
                        user:     "bob",
                        pass:     "bobsPassword1234",
                        isAuthed: true,
                        callnext: true,
                        rc:       200,
                },
                {
                        name:     "Given the User is NOT Authorized",
                        message:  "The next handler should **NOT** be called",
                        user:     "bob",
                        pass:     "bobsPassword1234",
                        isAuthed: false,
                        callnext: false,
                        rc:       http.StatusUnauthorized,
                },
        }

        for _, test := range tests {

                endpoint := fmt.Sprintf("https://%s:%s@some.endpoint/stuff", test.user, test.pass)

                req := httptest.NewRequest("GET", endpoint, nil)
                w := httptest.NewRecorder()

                t.Run(test.name, func(t *testing.T) {
                        mh := &MockHandler{t, test.message, test.callnext}
                        as := AuthMiddleware{
                                AuthService: func(user, pass string) bool {
                                        return test.isAuthed
                                },
                        }
                        as.Auth(mh).ServeHTTP(w, req)
                        if test.callnext {
                                assert.True(t, mh.wascalled, test.message)
                        } else {
                                assert.False(t, mh.wascalled, test.message)
                        }
                        assert.Equal(t, test.rc, w.Code, "Should get the expected return code")
                })

        }
}

 

Testing your Endpoint with httptest.Server

When testing with a server, you still start testing with the handler function. The biggest difference between using httptest.Server and httptest.ResponseRecorder is that Server starts up an actual server on a port.

func TestWithServer(t *testing.T) {

    ts := httptest.NewServer(http.HandlerFunc(EvenOrOddHandler))

    defer ts.Close()

    res, _ := http.Get(ts.URL + "?number=4")

    actual, err := ioutil.ReadAll(res.Body)
    assert.Nil(t, err)
    assert.Equal(t, `{"result:" "even"}`, actual)
}

 

You will find yourself using httptest.Server when you need to do end to end and or/integration testing. For example, you have a route that has a middleware that calls an external api/database, then does something before a response with another handler function. Using server would allow you to test the whole chain.
