Go’s net/http Package

If you’re coming to Go from another language, like: JavaScript, Python, Java, etc., you are probably familiar with using an external library or framework to stand up a web server. While these languages do offer core support for http servers, your code can quickly become verbose, brittle and difficult to maintain. Often leading us to look for viable solutions in the form of 3rd party libraries and frameworks.

Considering that Golang is relatively new compared to other languages. Coupled with the fact that web technologies have changed greatly in the past 10 years, Go’s core team was able to include excellent support for web services, right out of the box. That’s correct! Go’s net/http package is technically all you need to build a RESTful API.

Please note, there is nothing inherently wrong with using external libraries and frameworks. Golang was created to be a minimalist language, allowing you only pull in extra functionality when needed. To that end, the next few lessons will dig into building an API with the standard net/http package. As the lessons progress, and the API examples becomes more complex, additional packages will be introduced to show how to make the code more clean, robust, and manageable.

ListenAndServe

To start a server, only ListenAndServe is required.

package main

import (
	"fmt"
	"net/http"
)

func main() {
  fmt.Println("Starting server on port: 3001") //1
  http.ListenAndServe(":3001", nil)            //2
}
  1. Indicate that the server is running (optional)
  2. Call http.ListenAndServe passing a port as the first argument

The server can be started with: $ go run main.go.

please note: for demonstration purposes we’ll be using the fresh package: $ fresh main.go. Fresh is a command line tool that builds/restarts your web app on each save.

By using cURL, or navigating to http://localhost:3001/ in a browser, you’ll receive a 404 message.

curl http://localhost:3001
404 page not found

Don’t worry! This will be fixed momentarily. However, before moving on, take a look at the function signature for ListenAndServe

ListenAndServe Signature

func ListenAndServe(addr string, handler Handler) error

Breaking this down, you’ll see that ListenAndServe takes 2 arguments and returns an error

Arguments

  • addr: a string type, that indicates the port your app is running on
  • handler: a Handler type that handles incoming requests

Return

Because an error is returned, it is fairly common practice to wrap ListenAndServe in Log.Fatal to log any potential errors. In this case, log.Fatal will return an exit code of 1.

log.Fatal(http.ListenAndServe(":3001", nil))

Why nil?

Looking at the snippet above, it’s fair to ask: why is nil passed as the second argument?

By design, the DefaultServeMux is used when no handler is provided. We’ll cover this in more detail shortly. However, before we move on, let’s create a working route!

Handling Requests

To create a functioning endpoint, you’ll need to create a handler function

func rootHandler(w http.ResponseWriter, r *http.Request) {
		fmt.Fprintf(w, "welcome to %s from OM!", r.URL.Path)
}

http’s ResponseWriter & Request

You may be asking yourself about the arguments supplied to our rootHandler. What are they? We’ll start by looking at the ResponseWriter.

Go’s ResponseWriter docs state:

A ResponseWriter interface is used by an HTTP handler to construct an HTTP response

The ResponseWriter interface contains 3 methods:

  • Header() Header - Returns a map of headers that will be sent with the response
  • Write([]byte) (int, error) - An implementation of io.Writer used to create a response for the client
  • WriteHeader(statusCode int) - Sends an HTTP response header with the provided status code

In our example, the ResponseWriter (represented by the w variable) handles generating a “welcome from OM!" response for the client.

When it comes to Request it’s worth noting that, while ResponseWriter is an interface, Request is a struct.

Go’s Request docs state:

A Request represents an HTTP request received by a server or to be sent by a client

This representation can be used gather information about the request, such as:

  • Headers
  • Request Body
  • URL info
  • etc.

At this point, If you’re saying: Okay, I get it, but why are we using a pointer?

It turns out that the Request struct is considerably large. Copying it would be an expensive operation. Instead, we can just point to the Request in memory and gather the necessary information.

Activating the Handler Function

To activate that handler at / we’ll use http.handleFunc in our main function.

func main() {
	http.HandleFunc("/", rootHandler)
	// ...
}

The two arguments supplied to HandleFunc are:

  • / - indicates the route we want to activate
  • rootHandler - the handler function that we want to register when our route is triggered

Now, cURLing or navigating to http://localhost:3001 should result in a welcome message! Yay! 

$ curl http://localhost:3001
welcome to / from OM!

All together, the current code is:

package main

import (
	"fmt"
	"log"
	"net/http"
)

func rootHandler(w http.ResponseWriter, r *http.Request) {
	fmt.Fprintf(w, "welcome to %s from OM!", r.URL.Path)
}

func main() {
	http.HandleFunc("/", rootHandler)
	fmt.Println("Starting server on port: 3001")
	log.Fatal(http.ListenAndServe(":3001", nil))
}

Considering that we now having a working application, let’s dig into the handlers to gain a better understanding.

Handler & Handler Functions

Handlers are responsible for managing the request/response cycle. To aid in this effort, Go gives us the Handler interface, as well as a couple of methods that are useful for handling incoming HTTP requests.

The Handler Interface

The Handler itself, is an interface supplied by the net/http package.

The source for Handler is as follows:

type Handler interface {
        ServeHTTP(ResponseWriter, *Request)
}

According to the docs on Handlers:

A Handler responds to an HTTP request.

ServeHTTP should write reply headers and data to the ResponseWriter and then return…

It’s worth noting that any type can be a handler, so long as it satisfies the http.Handler interface. This can be accomplished by providing a ServeHTTP method. The handler can then be passed to ListenAndServe, which will call the ServeHTTP method on each incoming request.

Handle

According to the documentation on Handle:

Handle registers the handler for the given pattern…

Handle's function signature is as follows:

func Handle(pattern string, handler Handler)

We can see that Handle takes two arguments

  • pattern - a string type with the pattern (path) that you want to match (ie: /, /foo, /bar)
  • handler - a type that satisfies the Handler interface

Let’s refactor the previous example code to use this pattern (shortened for clarity).

package main
// ...

type rootHandler struct{} //1

func (rh rootHandler) ServeHTTP(w http.ResponseWriter, r *http.Request) { //2
		fmt.Fprintf(w, "welcome to %s from OM!", r.URL.Path)
}

func main() {
  rh := rootHandler{}  //3
  http.Handle("/", rh) //4

	// ...
}

Several Modifications were required:

  1. Create a rootHandler struct
  2. Re-write function to be a method on the rootHandler object and implement the ServeHTTP method (satisfying the Handler interface)
  3. Instantiate roothandler object as rh
  4. Call http.Handle passing in the path of / and the rh Handler

Running this code, will result in very similar output as our first example.

While this is extremely powerful, it’s a bit much for our simple web-server. So we’re going to look at the previously used HandlerFunc method

HandlerFunc

According to the HandlerFunc documentation:

The HandlerFunc type is an adapter to allow the use of ordinary functions as HTTP handlers.

The documentation goes on to say:

If f is a function with the appropriate signature, HandlerFunc(f) is a Handler that calls f.

This indicates that the HandlerFunc is essentially a helper function, that wraps a normal function with the func(http.ResponseWriter, *http.Request) signature, and returns a Handler

Let’s look at an example:

package main
// ...

func rootHandler(w http.ResponseWriter, r *http.Request) {  //1
	fmt.Fprintf(w, "welcome to %s from OM!", r.URL.Path)
}

func main() {
  http.Handle("/", http.HandlerFunc(rootHandler))           //2
  // ...
}
  1. Create a function with the correct signature
  2. Pass the function to your http.HandlerFunc

Behind the scenes, HandlerFunc attaches the ServeHTTP method to the function in question, therefore satisfying the Handler interface.

HandleFunc

While the name is very similar, HandleFunc has a slight difference.

The documentation for HandleFunc states:

HandleFunc registers the handler function for the given pattern

Before moving forward, take a look at the difference between the signatures of Handle and HandleFunc.

Handle

func Handle(pattern string, handler Handler)

HandleFunc

func HandleFunc(pattern string, handler func(ResponseWriter, *Request))

The major difference is that Handle requires a Handler type, while HandleFunc only requires a function with the func(ResponseWriter, *Request) signature.

With that in mind, HandleFunc takes the following arguments:

  • pattern - A string type with the pattern (route/endpoint)
  • func(ResponseWriter, *Request) - a function that makes use of http’s ResponseWriter interface, and Request struct.

If we look back to our original code sample, you’ll see this in action.

package main
// ...

func rootHandler(w http.ResponseWriter, r *http.Request) {
		fmt.Fprintf(w, "welcome to %s from OM!", r.URL.Path)
}

func main() {
  http.HandleFunc("/", rootHandler)  //1
	// ...
}
  1. Call HandleFunc passing the rootHandler function.

What should I use?

If you’re wondering whether to use HandleFunc or HandlerFunc consider these points.

  • HandleFunc is a nice shortcut that registers a function and let’s you handle requests
  • HandlerFunc makes an actual http.Handler that could be used in various cases. ie: middleware

404

Currently, any URI provided to the application will work. For example, you can navigate to http://localhost:3001/foo/bar or http://localhost:3001/hello/world and data will be rendered.

This immediately poses the question: how can we render valid routes only, and serve a 404 response when the route is invalid?

At the moment, all routes contain a / character, which means that there will always be a match. A quick fix is to inspect the URL.Path in the rootHandler

// ...
func rootHandler(w http.ResponseWriter, r *http.Request) {
	if r.URL.Path != "/" {
		fmt.Fprintf(w, "The resource you're looking for cannot be found")
		return
	}
	fmt.Fprintf(w, "Landing Page!")
}
// ...

Alternatively, we can use the NotFound method from net/http

// ...
func rootHandler(w http.ResponseWriter, r *http.Request) {
	if r.URL.Path != "/" {
		http.NotFound(w, r)
		return
	}
	fmt.Fprintf(w, "Landing Page!")
}
// ...

Now, all requests that don’t match a given route supplied by a handler will return a Not Found message.

Subtree & Fixed paths

Two types of URL patterns exist when using the standard net/http package.

  • Fixed Paths - ie: /widgets, /widgets/new, /foo etc.
  • Subtree Paths- ie: /products/, /bar/, etc

Before digging in to the major difference, we’ll create another route handler.

func omHandler(w http.ResponseWriter, r *http.Request) {  //1
	fmt.Fprintf(w, "Hello from Orange Academy")
}

func main() {
	http.HandleFunc("/", rootHandler)
  http.HandleFunc("/om", omHandler)                        //2
	// ...
}
  1. Create the route handler
  2. Trigger the handler when the request URI matches "/om”

Fixed paths are pretty common in web development. The route is only triggered with an exact match. Navigating to http://localhost:3001/om should result in the Hello from Orange Academy message because there is an exact match.

Subtree paths end with a trailing /, and act similar to a wildcard (*) operator. Unless there is a specific handler for a given route, any request containing the matching URI (at the beginning) will match and trigger the route handler.

For example, add a trailing / to om

http.HandleFunc("/om/", omHandler)

Any request beginning with /om should trigger the omHandler. Now, try a couple of routes

  • http://localhost:3001/om/go
  • http://localhost:3001/om/go/day1

As you can see, instead of getting a 404, the omHandler is invoked.

Mux

The net/http package includes an HTTP request Multiplexer, most often referred to as mux.

What is a Multiplexer?

This presents a question: What, exactly, is a Multiplexer?

According to the Multiplexing definition on Wikipedia:

multiplexing… is a method by which multiple analog or digital signals are combined into one signal over a shared medium.

In our context this can be easily translated to routing.

Meaning, the Go standard library gives us a router (mux) to handle incoming HTTP requests.

ServeMux

The multiplexer in net/http is known as the ServeMux type.

ServeMux is a struct that maps (routes) incoming requests to the appropriate HTTP handler in your application.

To create a locally scoped instance of ServeMux, we can call the NewServeMux function

mux := http.NewServeMux()

note: mux := &http.ServeMux{} may be used as an alternative.

Looking at the signature for NewServeMux() you’ll notice the return value is a pointer to ServeMux: func NewServeMux() *ServeMux.

By scrolling a little further in the docs, you’ll notice that ServeMux implements some familiar methods:

  • Handle
  • HandleFunc
  • ServeHTTP

This means, after creating the ServeMux object, the handlers can be registered by swapping out http.HandleFunc() in favor of mux.HandleFunc().

// ...
	mux := http.NewServeMux()
	mux.HandleFunc("/", rootHandler)
	mux.HandleFunc("/om/", omHandler)
// ...

Now, the locally scoped instance of ServeMux is set to handle requests to / and /om/.

Unfortunately, after spinning up the server, you’ll only get 404 errors. :man_facepalming:

What went wrong?

Earlier, while setting up ListenAndServe, nil was supplied as the second argument: http.ListenAndServe(":3001", nil).

The method signature for ListenAndServe indicates that the second argument is of type Handler. This leads to the question: what qualifies as a type of Handler?

In order to satisfy the Handler interface, your type must implement the ServeHTTP method. Thankfully, ServeMux implements the the ServeHTTP method, therefore satisfying the Handler interface.

Meaning, the ServeMux object (mux) can be passed to ListenAndServe, and the application should behave as expected.

http.ListenAndServe(":3001", mux)

Now, restarting the server, our routes should work as expected.

Before moving on, let’s review the updates to our main function.

// ...
func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("/", rootHandler)
	mux.HandleFunc("/om/", omHandler)

	fmt.Println("Starting server on port: 3001")
	log.Fatal(http.ListenAndServe(":3001", mux))
}
DefaultServeMux

This brings us back to the question, what happens when nil is passed to ListenAndServe?

When using nil, Go provides a global instance of ServeMux, also known as DefaultServeMux.

The DefaultServeMux is instantiated on application startup and provided as the default Handler to ListenAndServe. Which explains why the application failed to operate as intended after creating a local instance of ServeMux.

In this situation, once the handlers were registered with mux, the DefaultServeMux was no longer aware of them.

While it’s pretty nifty that a Default Mux is provided, it’s not recommended for use in production applications. This is due to the fact that DefaultServeMux is a global variable, making it potentially available to third-party packages used in your project. If one of those packages were to be compromised, they could potentially use the DefaultServeMux to expose your application to vulnerabilities. Instead, it’s best to opt for using a locally scoped ServeMux.

Setting a Port

Currently, the port is hardcoded to :3001, while this works, a better practice is to set the port as environment variable. This allows for more flexibility in development, and helps with the transition to deployment.

First, add the following in main()

func main() {
  var port string                               //1
  if port = os.Getenv("PORT"); port == "" {     //2
    port = "3001"
  }

  // ...

  fmt.Println("Starting server on port:", port) //3
  log.Fatal(http.ListenAndServe(":"+port, mux)) //4
}
  1. Assign an empty string to the port variable
  2. Use os.GetEnv() to check for a PORT environment variable, and ensure it is not an empty string
  3. Inform the user about the port currently in use
  4. Replace the hardcoded value with the port

Testing the server should yield the same results.

To export a PORT environment variable, run: $ export PORT=<PORT_NUMBER>.

Try it with port 5000.

$ export PORT=5000

Starting the server, the application will now run on port 5000.

Summary

Go’s standard library provides everything needed for standing up a web-server. In fact, there are many applications in production that exclusively utilize the net/http package. Because net/http is a part of the standard library, you can rest assured that the code is well-maintained, tested, and reliable.

With that in mind, there are limitiations around only using the standard net/http pacakge. As you continue to build web applications and services, the time may come to look for alternative packages that offer more flexibilty and ease of use.

As always, it’s important to choose the right tool for the job, thankfully Go has provided excellent built-in tooling.
