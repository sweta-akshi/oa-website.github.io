2. Introduction

Now that you have mastered Redux, it’s time to talk about integrating Redux with React. For this lesson we will be using the Redux documentation and building the example todo app. We will also be using the CSS Styling from Todo MVC to make our Todo App look pretty.

Figure 1. Todos App

3. Project Setup

We’ll start by creating a new app and installing redux as well as the react binding for redux (called react-redux).

$ create-react-app redux-todo
$ cd redux-todo
$ yarn add redux

3.1. TodoMVC Styling

To make our TODO app look nice, we are going to use the CSS styles from Todo MVC. We will also be using the classnames library for conditionally setting className values on our components. You can install these dependencies via:

$ yarn add todomvc-common
$ yarn add todomvc-app-css
$ yarn add classnames

We can import the CSS files in the src/index.js file:

src/index.js
import React from 'react';
import ReactDOM from 'react-dom';
import './index.css';
import 'todomvc-common/base.css';       // add this line
import 'todomvc-app-css/index.css';     // add this line too
import App from './App';
import * as serviceWorker from './serviceWorker';

3.2. react-redux

In order to use Redux with React, we also need to install the react-redux package:

$ yarn add react-redux

This package provides special Redux bindings for use with React by giving us a couple of simple utility methods/components. One such utility component is the <Provider> component. Typically, this component wraps our entire application (at the root) and provides access to the store.

Let’s go ahead and wrap our application in <Provider>. This time we will put the <Provider> wrapper in our app.js file (instead of the index.js file).

app.js
import React from 'react'
import { Provider } from 'react-redux'      (1)
import store from './redux/store';          (2)
import './App.css'

function App() {
    return (
        <Provider store={store}>            (3)
            <section className="todoapp">
                <header className="header">
                    <h1>todos</h1>
                </header>
            </section>
        </Provider>
    )
}

export default App
1 Import the Provider component
2 Import the store from a local file we will be creating next
3 Nest your <Provider> component inside the <App> component.

4. Actions

Now, let’s go ahead and set up our actions. We can define our action’s in src/actions/index.js.

We’ll start by defining our action types. Edit src/redux/actionTypes.js and insert the following code:

src/redux/actionTypes.js
export const TOGGLE_TODO = 'TOGGLE_TODO';
export const SET_FILTER = 'SET_FILTER';

Here were are setting constants that are equal to string values. While not necessary for an application of this size, it is still recommended to follow the above convention.

Defining your action type’s as constants makes your code more robust and self-documenting.

Next we’ll need to add any other constants that might be applicable. In this case, we need to add our Visibility Filters that will allow the user to toggle between viewing all todos, just the completed todos, or just the incomplete todos.

Let’s create a src/constants.js file to hold our Visibility Filters values:

touch src/constants.js

Open that file in your editor and insert the following code:

src/constants.js
export const VISIBILITY_FILTERS = {
    ALL: 'all',
    COMPLETED: 'completed',
    INCOMPLETE: 'incomplete'
};

Finally, we’ll need to define and export our action creator functions:

src/redux/actions.js
import { TOGGLE_TODO, SET_FILTER } from "./actionTypes";

export const toggleTodo = id => ({
  type: TOGGLE_TODO,
  id
});

export const setFilter = filter => ({
    type: SET_FILTER,
    filter
});

5. Reducers

As we previously learned, A reducer is a pure function that takes the previous state, as well as an action, and then returns the next state. Before we can get too deep into the Reducers themselves, it would be a good idea to think about what our state will look like. Remember the state is going to be a single object. Let’s think about the structure of our state object and it’s key/values.

TODO State:

  • A todos property - array a list of todo objects

  • A visibilityFilter property - string representing the current filter the user has selected

Each todo object will have 3 properties:

  • id - a numeric (integer) id for the todo

  • title - string indicating the the title of the todo

  • completed - boolean value indicating whether the task is complete

Example State
{
  visibilityFilter: 'all',
  todos: [
    {
      id: 1,
      title: 'Consider using Redux',
      completed: true,
    },
    {
      id: 2,
      title: 'Keep all state in a single tree',
      completed: false
    }
  ]
}

Now, we need to teach our reducers how to process the actions they will be receiving. First let’s define our initial state and define a single reducer to handle all of our actions.

Replace the code in src/redux/reducers/index.js with the following code:

src/redux/reducers/index.js
import { SET_FILTER, TOGGLE_TODO } from "./actionTypes";
import { VISIBILITY_FILTERS } from "./constants";

const initialState = {
    filter: VISIBILITY_FILTERS.ALL,
    todos: []
}

const rootReducer = (state = initialState, action) => {
    switch (action.type) {
        case SET_FILTER: {
            return {
                ...state,
                filter: action.filter
            }
        }
        case TOGGLE_TODO: {
            const todos = state.todos.map(todo => todo.id === action.id ?
              { ...todo, completed: !todo.completed } : todo)
            return {
                ...state,
                todos
            }
        }
        default: {
            return state;
        }
    }
}

export default rootReducer;

At this point we are focusing on just 2 actions, SET_FILTER and TOGGLE_TODO.

When the state is initially loaded (connected / subscribed) our reducer should return an initial state of:
{visibilityFilter: 'all', todos: []}.

Once a SET_FILTER action is dispatched describing an update to the visibilityFilter, the reducer will:

  1. see the action 's payload indicating the new filter value (ALL, COMPLETED, or INCOMPLETE)

  2. return a new state object with an updated visibilityFilter value.

5.1. Reducer Composition

Taking a look at our existing code you can see that the reducer can become quite long. Just consider trying to support a dozen or more actions! Perhaps it’s time to consider a bit of refactoring, specifically splitting the reducer into multiple reducers. But how should we split our reducer?

Let’s consider that our store has 2 major properties.

  • visibilityFilter

  • todos

So it makes sense to split the reducer into multiple reducers where each reducer is only concerned about a particular slice of the state. This pattern is known as reducer composition, and it is used extensively in Redux applications.

5.1.1. The visibilityFilter Reducer

To start, let’s create a reducer function for the visibility filter slice of the state. Our current code for the visibility filter looks like this…​

before
const rootReducer = (state = initialState, action) => {
    switch (action.type) {
        case SET_FILTER: {
            return {
                ...state,
                filter: action.filter
            }
        }
      // ...
    }
  }

Todos will follow a similar pattern. In this case, the slice of state that we want to work with is the todos array. This will require some small, but important changes to the code.

5.2. The Root Reducer

The root reducer will handle calling our smaller reducer functions and composing the values. Writing our root reducer is very simple.

src/redux/reducers/index.js
import { combineReducers } from "redux";
import visibilityFilter from "./visibilityFilter";
import todos from "./todos";

export default combineReducers({ todos, visibilityFilter });
The Redux library provides the handy combineReducers function to help us combine multiple reducers into a single root reducer. combineReducers takes your reducer functions as arguments and returns a reducer function that invokes every function passed to it. It then constructs a state object to reflect the new state.

This type of composition and structure allows us to write composable functions each with a single responsibility of managing their own slice of the state. In larger applications this makes your code much cleaner and easier to follow.

7. Container Components

Now, we can create the container components and hook them up the the store. Before we proceed, there are a few items in the Implementing Container Components section of the Redux docs that are worth note.

Technically, a container component is just a React component that uses store.subscribe() to read a part of the Redux state tree and supply props to the presentational component it renders.

No magic here, our container components are the ones that subscribe to the store, requesting the piece of state it is concerned about. That component then passes the data down to it’s children/presentational components as props.

To connect our container components to the Redux store, we will be using the connect() function provided by react-redux

From the Redux documentation:

Technically you could write the container components by hand using store.subscribe(). We don’t advise you to do this because React Redux makes many performance optimizations that are hard to do by hand. For this reason, rather than write container components, we will generate them using the connect() function provided by React Redux…​

— Redux Documentation

The connect() function will take care of generating our component (HoC style) and return a component that is subscribed to the store. The authors of Redux have gone to great length to make the connect() components as performant as possible.

Let’s take a look at some code to get a better understanding.

The FilterLink component will be responsible for connecting to the Redux store, getting the

7.1.1. connect()

Before moving on, we need to take a look at connect(). If the syntax here: const FilterLink = connect()(Link) looks foreign, or strange, just know that you’re not alone. To understand what’s happening, we first need to look at the return value of connect.

According the to react-redux documentation, The connect() function returns…​

A higher-order React component class that passes state and action creators into your component derived from the supplied arguments.

— react-redux docs

But what does that even mean?

connect returns a function that is a Higher Order Component factory. This factory is able to then create a Higher-Order Component (an HOC) that wraps your component so that it will receive the proper state and actions as props.

So when you call connect you get a Higher Order Component factory. Then you pass your component as an argument to that function to get the final HOC. If you’re familiar with currying, this is very similar. The entire process will generate a component that is connected / subscribed to the store.

In this case the HOC is named FilterLink. That’s a lot to think about and it will start to make more sense once we look at connecting it with the store’s state via mapStateToProps and mapDispatchToProps, which we will look at next.

7.1.2. mapStateToProps

Now that we understand a bit more about how connect works, we need to understand how it passes state (and action creators) from the store to the HOC.

Looking at the documentation, you’ll notice that connect 's function signature can take up to 4 arguments. The first is mapStateToProps, which is a special function that transforms state (from the store) into props for our presentational component. Note that:

  • mapStateToProps is called anytime the store is updated

  • the return value must be a plain object

  • the return object will be merged into the component’s props

More about mapStateToProps can be found in arguments section of the react-redux documentation on connect.

7.1.3. mapDispatchToProps

So, now we know how to send data from the store to our components, but how do we send our action creators? Thankfully, the connect function takes a second special argument named mapDispatchToProps that will help us out. mapDispatchToProps is a function that receives the dispatch function (as an argument) and returns functions that can be passed as props. This way we don’t have to pass dispatch and action creators down manually.

7.1.4. FilterLink Revisited - Putting It All Together

Let’s determine how we can use mapStateToProps in our first container component, FilterLink:

src/components/FilterLink.js
import { connect } from 'react-redux';
import Link from './Link';                                  (1)
import { setFilter } from './redux/actions';               (2)

const mapStateToProps = (state, ownProps) => {              (3)
  return {
    active: ownProps.filter === state.visibilityFilter      (4)
  };
};

const mapDispatchToProps = (dispatch, ownProps) => {        (5)
  return {
    onClick: () => {                                        (6)
      dispatch(setFilter(ownProps.filter));                 (7)

  };

export default FilterLink;

2 Import the Link component
3 receive state and props (see below for explanation) as arguments
4 compare the props.filter value to the state.visibilityFilter value to determine if this Link is active
5 receive dispatch and props as arguments
6 onClick dispatch the setFilter action creator that will describe an intention to change the current visibilityFilter
ownProps is the second argument that can be passed to mapStateToProps and mapDispatchToProps and represents the props being passed to the container component. By convention it is named ownProps.

7.3. VisibleTodoList

The point of this component is to only render the todo items associated with the current filter chosen by the user. For example, when the user clicks on the completed link, all incomplete tasks should be filtered out.

To do this, we can write a simple function that takes in the todos array and user selected filter and return an array with the proper contents.

Create the file src/components/VisibleTodoList.js and insert the following code:

src/components/VisibleTodoList.js
import { VISIBILITY_FILTERS } from './constants'

const getVisibleTodos = (todos, filter) => {         (1)
    switch (filter) {                                (2)
        case VISIBILITY_FILTERS.COMPLETED:
            return todos.filter(t => t.completed);   (3)
        case VISIBILITY_FILTERS.INCOMPLETE:
            return todos.filter(t => !t.completed);  (4)
        case VISIBILITY_FILTERS.ALL':
        default:
            return todos;                            (5)
    }
};
1 pass in todos and filter as arguments
2 write a switch statement that evaluates the filter value
3 if completed, return an array with only complete tasks
4 if active, return an array with only incomplete (active) tasks
5 otherwise return all todos

Now we can use the store’s state to determine which set of todos should be passed down as props to the TodoList presentational component.

The getVisibleTodos function is sort of like a reducer function. But in this case we don’t need to connect this function to our reducers / store because we are simply doing the calculation locally in the VisibleTodoList component and don’t need to manage the list of visible todos in the store (we will just render them instead).

To make all of this work, we need to define a mapStateToProps function to get all of the todos from the store and then pass them to the getVisibleTodos function to filter them based on the store’s filter value. We also need to define a mapDispatchToProps funtion for passing any callbacks as props where the callback needs to dispatch an action to the store.

Here is our updated VisibleTodoList code:

src/components/VisibleTodoList.js
import { connect } from 'react-redux';
import TodoList from './TodoList';
import { toggleTodo } from './redux/actions';
import { VISIBILITY_FILTERS } from './constants';

const getVisibleTodos = (todos, filter) => {
    switch (filter) {
        case VISIBILITY_FILTERS.COMPLETED:
            return todos.filter(t => t.completed);
        case VISIBILITY_FILTERS.INCOMPLETE:
            return todos.filter(t => !t.completed);
        case VISIBILITY_FILTERS.ALL:
        default:
            return todos;
    }
};

const mapStateToProps = state => ({
    todos: getVisibleTodos(state.todos, state.visibilityFilter)
})

const mapDispatchToProps = dispatch => {
    return {
        toggleTodo: id => {
            dispatch(toggleTodo(id));
        }
    }
};

const VisibleTodoList = connect(mapStateToProps, mapDispatchToProps)(TodoList);

export default VisibleTodoList

7.3.1. The Object Shorthand Form of mapDispatchToProps

There is an alternative implementation of mapDispatchToProps that is available under certain conditions. For example, the above mapDispatchToProps function can be replaced with the following single line of code:

const mapDispatchToProps = { toggleTodo }

which is simply an object that maps prop names to actionCreators. Here we are also taking advantage of the fact that our prop names and our action creators are spelled the same. The object form of mapDispatchToProps is more generically the following:

const mapDispatchToProps = {
    propCallbackA: actionCreatorA,
    propCallbackB: actionCreatorB
}
Be aware that this shortcut is only available when the callbacks take the exact same parameters as the action creators. For more information see Two Forms of mapDispatchToProps.
